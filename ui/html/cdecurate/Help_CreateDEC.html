Name a Data Element Concept (NEW FEATURES)
The ISO 11179 Standard describes a Data Element Concept as the combination of an Object Class, a Property, and Qualifiers for each. Definitions for each of these terms are documented below.
What is a Data Element Concept?
The ISO/IEC 11179 Standard describes a Data Element Concept as relating several data elements that record data about that concept with different representations. For example, a concept of PatientPlaceOfBirth is a Data Element Concept.

A Data Element Concept is composed of an Object Class, a Property and optional Qualifiers for both.
What is an Object Class?
The ISO 11179 Standard describes an Object Class as ‘a set of ideas, abstractions, or things in the real world that can be identified with explicit boundaries and meaning and whose properties and behavior follow the same rules’. In other words, Object Classes are the ‘things about which you want to store and collect data.’ Examples of Object Classes include cars, persons, households, employees, etc. For the example of PatientPlaceOfBirth, an Object class is Patient.
What is a Property?
The ISO 11179 Standard describes a Property as ‘a peculiarity common to all members of an object class’. In other words, properties are ‘what humans use to describe or distinguish Objects’. For the example given above, a Property is PlaceOfBirth.
What are Qualifiers?
The ISO 11179 Standard describes qualifiers as being used to ‘differentiate between names (i.e. Object Class and Property combinations) that would otherwise be the same’. For the example of PatientPlaceOfBirth, a Qualifier of year is used to indicate a specific year.

Data Element Concepts are named by combining Primary and Qualifier Concepts for Object Classes and Properties. The Object Classes and Properties are then used for naming the Data Element Concept. This approach is new for version 3.0 of the CDE Curation Tool. Building Blocks for Object Classes and Properties

Object Classes and Properties are the building blocks from which a Data Element Concept name is made. When creating a new Data Element Concept the use of Object Classes and Properties is strongly encouraged but not strictly mandatory. The CDE Curation Tool enables you to search caDSR and the NCI EVS (Enterprise Vocabulary Services) for for Concepts. The Primary Concepts may represent existing Object Classes and Properties, or Concepts may be combined to create new Object Classes and Properties. Figure 4 shows the interface to search for Concepts for Object Classes and Properties. Object Class and Property Long Names are also created from this interface.

An Object Class or Property may consist of one Primary Concept and one or more Qualifier Concepts. Each of these Concepts must be selected separately. Once selected the Concept's source database and Concept Code will be shown below the Concept. You may highlight Qualifier Concepts one at a time to see their source database and Concept Code.

The Object Class and Property Long Names will be constructed from the names of the Qualifier and Primary Concepts that have been chosen. The Long Name will be built left to right starting with the Concept at the top of the Qualifier Concepts list and ending with the Primary Concept. Once you have built the Object Class and Property Long Names they are not editable.

Please note that if you select only a Primary Concept from the caDSR database and select no Qualifier Concepts you have in effect selected an existing Object Class (or Property). In this case, no new Object Class (or Property) will be created in caDSR when you submit your Data Element Concept.

When the Tree Search first appears all of the NCI Thesaurus Root Concepts will appear on the left side of the screen. Each Root Concept may have one or more Subconcepts. A Concept's Subconcepts may be seen by clicking on the plus sign next to the Concept name. Some Subconcepts will have Subconcepts of their own which may be seen by clicking on a plus sign that appears next to the Subconcept name. Verify Long Name and Preferred Name
The CDE Curation Tool automatically generates a suggested drafts of the Data Element Concept Long Name. This suggestion is made up of the building blocks you selected in the previous step. The Long Name field will contain the full Long Names of the selected Object Class and Property. If no Object Class or Property were selected then you must type in a Data Element Concept Long Name.

There are three options for creating a Data Element Concept Preferred Name. It is strongly suggested that you accept the default System Generated Preferred Name. This name will be a combination of the Object Class Public ID and Version with the Property Public ID and Version, although the Preferred Name field will simply say ‘(Generated by the System).’ If you want you may select an ‘Abbreviated’ Preferred Name. This Preferred Name type will consiste of the first four letters of each building block Concept selected for the Object Class and the Property, separated by the character ‘_’. If the Prefered Name is longer than 30 characters it will be truncated to 30 characters. The final Preferred Name option is ‘Existing Name (editable).’ When you choose this otion you type a Preferred Name into the field, up to the 30 character limit. If you selected no Object Class or Property you must use an ‘Existing Name’ type of Preferred Name. Figure 10 illustrates the populated Data Element Concept Long Name and Preferred Name.

Search for a Conceptual Domain
What is a Conceptual Domain? The ISO 11179 Standard describes Conceptual Domain as ‘a set of all possible, valid value meanings of a data element concept expressed without representation’. Conceptual Domains are plural, object oriented, lacking in representation, and may be associated with more than one Data Element Concept. For the example given above with the Data Element Concept of PatientPlaceOfBirth, a Conceptual Domain is Country.

When the Search link is activated, all Conceptual Domains in the selected Context are displayed. The user has the option to change the Context on the Search screen. The records will change accordingly.

If no context has been selected, no results will be displayed and the user will have to do at least one of the following actions:
  1. Click Start Search (all Conceptual Domains for all Contexts will be displayed)
  2. Enter a Search Term and Select a Context, then click Start Search (results matching the term and context will be displayed)
  3. Select a Context then click Start Search (all Conceptual Domains for the selected Context will be displayed

Select a Workflow Status
A Workflow Status is a designation of the position in the lifecycle of a Data Element Concept. Select a Data Element Concept’s Workflow Status by choosing from the list of available Workflow Statuses in the drop down list. Many Workflow Statuses are listed and defined on the NCI Center for Bioinformatics Business Rules page. The Versioning Business Rules are also accessible from within the CDE Curation Tool by clicking on the ‘Business Rules’ hyperlink next to the Version field. This is a new feature for version 3.0 of the CDE Curation Tool.

Please note that a Data Element Concept cannot have a Workflow Status of ‘Released’ unless its Object Class and Property also have Workflow Statuses of ‘Released.’ It should be a very rare for this to be a problem. However, if it does occur you must update the Object Class and Property Workflow Statuses using the caDSR Administration Tool as there is no mechanism for changing the Object Class and Property Workflow Statuses in the CDE Curation Tool. Enter/Select the Effective Begin and End Dates
The Effective Begin Date is mandatory, while the Effective End Date is not. Effective Begin Date will default to the current date. You may change it in two ways; either enter the date in the text box using the format MM/DD/YYYY or click on the calendar icon and select the date. Figure 17 illustrates the steps required for entering or selecting an Effective Begin Date for a Data Element Concept.

You may select or enter an Effective End Date in exactly the same way if you wish. If you have selected a Workflow Status of ‘Retired Archived,’ ‘Retired Deleted,’ or ‘Retired Phased Out’ then you must enter an Effective End Date. Note also that the Data Element Concept validation process will return an error if you select an Effective End Date that is before the Effective Begin Date.

Select Classification Scheme/Classification Scheme Items (NEW FEATURES)
Classification Schemes are a way of grouping Data Element Concepts based on common characteristics. Classifications also help add information about Data Element Concepts that is not easily included in definitions. For example, you might want to classify Data Element Concepts as belonging to the Classification Scheme Type of Disease. A Classification Scheme Item is a component of content in a Classification Scheme (i.e. Liver Cancer).

The parent-child relationships of Classification Scheme Items are displayed in a hierarchical order and also by indentation. Child Classification Scheme Items are indented farther than their parent. This is a new feature for version 3.0 of the CDE Curation Tool.

Versioning (includes NEW FEATURE)
To change the version for a group of Data Element Concepts:
  • Check the box labeled "Check Box to Create New Version"
  • Check either the box labeled "Point Increase" or "Whole Increase"
Point Increase: Increases the version number of each Data Element Concept in the group by a tenth. (i.e. If your group of Data Element Concepts have versions of 4.0 and 1.23 the version would be changed to 4.1 and 1.3 respectively).

Whole Increase: Increases the version number of each Data Element Concept in the group to the next whole number (i.e. If your group of Data Element Concepts have versions of 4.0 and 1.5 the version would be changed to 5.0 and 2.0 respectively). Figure 37 illustrates this process.

NOTE: All edits made during a block edit containing version editing will be made to the new versions of the selected Data Element Concepts. The edits will not be made to the originally selected Data Element Concepts. A list containing all versioning rules within the caDSR can be found at the NCI Center for Bioinformatics caDSR versioning page. The Versioning Business Rules are also accessible from within the CDE Curation Tool by clicking on the ‘Business Rules’ hyperlink next to the Version field. This is a new feature for version 3.0 of the CDE Curation Tool.
